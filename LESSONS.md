# 金桥系统 — 经验教训沉淀

> 最后更新: 2026-06-17 | 累计 v1 → 现在

## 教训1: 数据管线不可靠是所有判断失准的根源

**日期**: 2026-06-17
**触发**: 收盘复盘5/9个股收盘价偏移1天，虚拟盘净值失真

**教训**:
- 数据源静默失效比显式报错更危险——脚本"成功执行"，输出"看起来正常"的数字，但数字是错的
- 下游（虚拟盘净值、触发条件、操作建议）全部依赖这些数字，一根管断、全链失准
- 修复不是加一个 fallback 就够，需要三重保障：时间戳校验 + 多源交叉 + 数据源溯源

**写入代码**: closing_review.py v2.6.0, generate_trade_signals.py v2.4.0, update_daily_cache.py v1.1

## 教训2: 中间层转发不可靠，最短路径最可靠

**日期**: 2026-06-17
**触发**: 价格穿越预警四轮迭代才修通

**教训**:
- v1: subprocess.run(openclaw message send) — isolated cron toolsAllow 拦截
- v2: print 到 stdout → agent 读取 — stdout 被 cron 环境吞掉
- v3: 写入文件 → agent 读取 → announce — announce 标记 delivered 但群聊无消息
- v4: 脚本内直接调钉钉机器人 API — 一次成功

每一层转发增加一个失败点。推送的最终目标是"消息出现在群聊"——直接在脚本里调机器人 API 就是这条路上的最短路径。

**写入代码**: price_watch.py（钉钉机器人 API 直推）

## 教训3: 重试机制不能掩盖静默失败

**日期**: 2026-06-17
**触发**: 价格预警 cron 的 30 分钟去重冷却 + 推送失败静默

**教训**:
- 脚本调了 push_alerts → 认为已推送 → 写了去重状态
- 实际消息被拦截，但去重状态已写，后续 30 分钟内不再尝试
- 结果：检测到了 2 条穿越，用户一条也没收到
- 去重应该在推送成功后才写入，而不是推送前

**写入代码**: price_watch.py（去重状态在 API 返回 200 后才写入）

## 教训4: 跨项目脚本路径依赖是定时炸弹

**日期**: 2026-06-17
**触发**: 外盘信号依赖 overseas-morning-brief 项目脚本

**教训**:
- 金桥 run_overseas_push.sh 引用 overseas-morning-brief/scripts/extract_signal.py
- 两个独立仓库，任一项目路径变更即断链
- 修复：关键脚本复制到金桥仓库内，输出文件改到金桥目录
- 外部项目只负责生成原始数据（morning_brief），金桥自包含处理+推送

**写入代码**: extract_signal.py → 金桥 scripts/, overseas_signal → 金桥 reports/

## 教训5: 历史数据错误不会自动修正

**日期**: 2026-06-17
**触发**: 虚拟盘净值历史需要回补修正

**教训**:
- 6/15~16 的虚拟盘净值基于过期收盘价计算，发现后无法回退
- 修正代码只能保证"从今天起正确"，不修正历史
- 正确做法：发现数据问题后，除了修代码，还要用正确数据重建受影响的历史记录
- **待执行**: 用腾讯+新浪交叉验证的真实收盘价，重算 6/15~17 虚拟盘净值

## 教训6: 错误价格比没有价格更危险

**日期**: 2026-06-17
**触发**: 复盘报告列出的"收盘价"其实是6/15的过期数据

**教训**:
- 如果当天拿不到收盘价，输出"无数据"比输出一个错误数字好
- 过期价格看起来像正常数字——有小数、能算出涨跌幅、能被下游消费——但全是错的
- 所有价格数据源必须附带时间戳校验，日期不匹配的不进入计算

**写入代码**: closing_review.py（新浪 fields[30] ≠ 今天 → 跳过，降级到腾讯）

## 通用原则（从教训提炼）

1. **数据管线必须有端到端交叉校验**，单源+静默接受=迟早偏移
2. **推送走最短路径**，不经过中间层转发
3. **去重/缓存状态在操作成功后写入**，不在操作前
4. **所有外部依赖内化到项目仓库**，跨项目引用是事故温床
5. **发现数据问题后修正历史记录**，不只是修代码
